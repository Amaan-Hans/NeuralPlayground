Your Brain Has a GPS

How a Nobel Prize-winning discovery became a Python framework

A Discovery Worth a Nobel Prize

Somewhere inside your skull, specialized neurons called grid cells fire in a perfect hexagonal pattern as you move around a room — like invisible dots on a map. Scientists won a Nobel Prize for discovering this in 2014. NeuralPlayground is the code that lets researchers simulate and test theories about how these neurons work.

These grid cells work alongside place cells in the hippocampus to form a complete internal map of any environment. Together they are the biological GPS that every mammal carries — and NeuralPlayground is the simulator for this system.

💡
Key Insight

NeuralPlayground is a research framework — it gives scientists a standardized way to run virtual experiments, swap out brain models, and compare results against real neural recordings from live animals.

Reading the Training Loop

The training loop in backend/training_loops.py is the heartbeat of every experiment — it runs thousands of times per training session. Here is exactly what each line does in plain English.

CODE 
obs, state = env.reset()
training_hist = []
obs = obs[:2]
for j in range(round(n_steps)):
    # Observe to choose an action
    action = agent.act(obs)
    # Run environment for given action
    obs, state, reward = env.step(action)
    update_output = agent.update()
    training_hist.append(update_output)
    obs = obs[:2]
PLAIN ENGLISH

obs, state = env.reset() — Set up the arena from scratch: randomize starting position, clear all history

obs = obs[:2] — Keep only x,y coordinates (ignore extra sensor data for now)

action = agent.act(obs) — Ask the brain model: given your current position, which direction do you want to move?

obs, state, reward = env.step(action) — Execute that movement: check for walls, update position, return new location

update_output = agent.update() — Let the brain model learn from what just happened: adjust internal connections

training_hist.append(update_output) — Log the learning statistics for later analysis

Meet the Cast

The four actors in every NeuralPlayground simulation

Every simulation has four players. Once you know their names and roles, you can tell an AI assistant exactly what to change and where — without getting lost in 5,000 lines of code.

Think of it as a stage play. There is a set, an actor, a script consultant, and a director. Each has exactly one job — and they can be swapped out without touching the others.

The Four Players

🏟️
Arena — The set

The virtual environment the agent moves through: a square room, a real-world maze layout, or any geometry you define. It knows about walls, positions, and rewards.

🧠
Agent — The actor

The brain model — a computational theory of how neurons learn to navigate. Weber2018, Stachenfeld2018, or any new model you bring in. Must follow the abstract class contract.

🔬
Experiment — The script consultant

Real rat neural recordings from published papers. Hafting2008, Sargolini2006 — actual spike times and positions from real animals used as scientific ground truth.

🎬

The orchestration layer that runs the training loop, coordinates the other three components, and logs results. It calls the shots so no one else has to.

The Heartbeat Loop

Four lines that run 20,000 times — and everything else supports them

Four Lines Rule the World

There is one piece of code that runs tens of thousands of times in every simulation. It is four lines long. Everything else in the framework exists to support those four lines.

A heartbeat has four phases: squeeze, pause, refill, relax. Miss one and the whole system stops. The reinforcement learning training loop has the same rhythm: observe, act, step, update. This four-beat pattern is universal — it appears in every RL system ever written.

💡

This exact four-beat loop — observe, act, step, update — appears in EVERY reinforcement learning system. OpenAI Gym, DeepMind Lab, robotics simulators. Learn this pattern once; read any RL codebase forever.

Every line in the core loop has one job. Here is exactly what each one does in plain English — from backend/training_loops.py, lines 29–41.

CODE

PLAIN ENGLISH

Reset the arena: randomize the starting position, clear all history from previous runs

Create an empty list to store learning statistics as training progresses

Keep only x,y coordinates — the arena returns extra data, but we only need position

Repeat this block for every step in the simulation (up to n_steps total moves)

Ask the brain model: given where you are, which direction do you want to move?

Execute that movement: check for walls, update position, return new location and reward signal

The brain model learns: adjust connection weights based on what it just experienced

Log the learning statistics for later analysis and debugging

Notice the observation space slice obs[:2] appears twice: once before the loop, once at the end of each iteration. The arena hands back more data than the agent needs — that trim keeps everything consistent.

The Four-Beat Rhythm

Each beat has a single owner and a single responsibility. The agent owns beats 1 and 2 and 4. The arena owns beat 3. Neither can skip its turn.

1
OBSERVE

The agent receives its current x,y position from the arena. This is all it knows — no memory of past positions, no knowledge of walls, just a two-number coordinate.

2
ACT

The brain model computes which direction to move. In the base agent, this is a random Gaussian step. In Weber2018, it uses learned weights shaped by the hippocampus model.

3
STEP

The arena executes the move: checks for walls via validate_action(), updates position, calculates reward signal, and returns the new state.

4
UPDATE

The brain model learns. It adjusts its internal connections based on where it just was and what it experienced. In episode terms, this is where the neurons wire together.

🔬
The Base Agent Does Almost Nothing

The agent.update() method in AgentCore returns None. That is intentional — it is a placeholder that subclasses like Weber2018 override with real learning algorithms. The loop structure stays the same regardless.

Where Do You Plug In?

When you ask AI to add a feature, the answer is almost always "put it in the loop — but where?" The answer depends on what information that feature needs.

before the loop One-time setup: load a checkpoint, open a log file, initialize a renderer. Runs once.
after agent.act() You have the action but not yet the result. Useful for visualizing intended movement before it happens.
after env.step() ★ Most common insertion point. You have the new position and reward — ideal for logging, visualization, early stopping, and progress checks.
after agent.update() You have the learning stats from this step. Good for logging loss curves or saving checkpoints at intervals.
after the loop Post-processing: plot trajectories, save the final model, run summary statistics. Runs once.

How Neurons Learn Space

Two theories, two algorithms — and how grid patterns emerge from nothing

Grid Patterns From Nothing

When a rat explores a room, specific neurons fire ONLY in specific spots — like having a dedicated alarm for each location. Two competing theories explain how this emerges from scratch. Both are in this codebase. Here is how they actually work.

The remarkable thing: nobody programs the hexagonal pattern. It begins with thousands of neurons connected by tiny random numbers. After enough training steps, a perfectly tiled grid appears — spontaneously. Two completely different algorithms produce it through completely different logic.

🧠
What "Emergence" Means Here

Emergence means the grid pattern is not stored anywhere in the code. It is a consequence of the learning rules, the number of neurons, and repeated exposure to a space. Change the learning rate and you get a different pattern. Change the arena shape and the grid adapts. The algorithm produces structure, but the structure itself is never specified.

Weber2018

Pure Hebbian learning. Excitatory and inhibitory neurons compete until spatial structure crystallizes. No explicit reward. No map. Just correlation.

Stachenfeld2018

The Successor Representation. Instead of learning where you are, it learns where you will likely be. Past experience shapes future prediction.

Weber2018: The Musician Metaphor

Imagine two musicians who jam together regularly. The more they play together, the stronger their musical bond — they anticipate each other, finish each other's phrases. In Weber2018, neurons work the same way: if two neurons fire at the same location repeatedly, the connection between them strengthens. That is Hebbian learning in one sentence.

The network has 4,900 excitatory neurons pushing activity up and 900 inhibitory neurons pulling it down. Their competition, repeated thousands of times, carves the space into distinct firing zones.

CODE — agents/weber_2018.py lines 375–419 
def update(self, exc_normalization: bool = True, pos: np.ndarray = None):
    if pos is None:
        pos = self.obs_history[-1]
    if len(pos) == 0:
        pass
    else:
        r_out = self.get_output_rates(pos)
        # Excitatory weights update (eq 2)
        delta_we = self.etaexc * self.get_rates(self.exc_cell_list, pos=pos) * r_out
        # Inhibitory weights update (eq 3)
        delta_wi = (
            self.etainh
            * self.get_rates(self.inh_cell_list, pos=pos)
            * (r_out - self.ro)
        )
        self.we = self.we + delta_we
        self.wi = self.wi + delta_wi
        if exc_normalization:
            self.we = (
                self.init_we_sum / np.sqrt(np.sum(self.we**2) + 1e-8) * self.we
            )
PLAIN ENGLISH

pos = self.obs_history[-1] — Get the most recent position from memory (default: the last visited location)

r_out = self.get_output_rates(pos) — Calculate how strongly the output neuron fires right now at this position

delta_we = self.etaexc * ... * r_out — Excitatory weight change = learning rate × input firing × output firing. Neurons that fire together, wire together.

delta_wi = self.etainh * ... * (r_out - self.ro) — Inhibitory weight change includes a target rate (ro). This prevents the network from firing everywhere ALL the time.

self.we = self.we + delta_we — Apply the update — strengthen the excitatory connections

self.wi = self.wi + delta_wi — Apply the update — strengthen the inhibitory connections

self.we = (self.init_we_sum / np.sqrt(...) * self.we) — Normalize so total weight strength stays constant — this prevents runaway growth

✨
Emergence is Magic

Nobody programs the hexagonal grid pattern into Weber2018. It emerges spontaneously from thousands of steps of simple weight updates. This is one of the most profound results in computational neuroscience.

Stachenfeld2018: The Subway Map Metaphor

Think of a subway map. When you stand at a station, the map does not just show where you ARE — it shows you where you can GO and how many stops away each destination is. Stachenfeld2018 builds exactly this kind of predictive map in the brain: each location encodes not just "I am here" but "here is where I am likely to be in the future."

This is the Successor Representation — a matrix updated by temporal difference learning every time the agent takes a step. The grid-cell-like patterns emerge from eigendecomposition of that prediction matrix.

CODE — agents/stachenfeld_2018.py lines 179–200 
def obs_to_state(self, pos: np.ndarray):
    diff = self.xy_combinations - pos[np.newaxis, ...]
    dist = np.sum(diff**2, axis=1)
    index = np.argmin(dist)
    curr_state = index
    return curr_state
PLAIN ENGLISH

diff = self.xy_combinations - pos[np.newaxis, ...] — For every grid point on the internal map, compute the vector from that point to the agent's current position

dist = np.sum(diff**2, axis=1) — Convert each vector to a scalar distance (squared, to avoid square roots — faster and mathematically equivalent for comparisons)

index = np.argmin(dist) — Find the grid point with the smallest distance — that is the nearest discrete state

return curr_state — Return that index number. A continuous x,y coordinate is now a single integer the SR matrix can use.

Why does this function exist at all?

The arena gives continuous x,y coordinates like [0.312, -0.087]. But the Successor Representation is a matrix indexed by integers — state 0, state 1, state 2. obs_to_state() is the bridge: snap the continuous position to the nearest point on a pre-built grid, return its index. Without this, the continuous world and the discrete math never connect.

Three Models, One Interface

NeuralPlayground ships with three brain models implementing three distinct theories of spatial cognition. Each uses different mathematics, different data structures, even a different framework — but all plug into the same training loop.

🧪

Weber2018

Hebbian plasticity. 4,900 excitatory + 900 inhibitory neurons. Grid cells and place cells emerge from pure correlation learning.

🗺️

Stachenfeld2018

Successor Representation. Builds a predictive map: where will I be in the future? Uses TD learning on a discretized state space.

🔗

Whittington2020

Tolman-Eichenbaum Machine. A graph neural network (PyTorch). Learns relational structure between locations. Requires discrete environments.

🔑
The AgentCore Contract

Three models, three completely different algorithms — but all share the same interface: act() and update(). This is why you can swap them with one line of code.

Decision-making. Given a position, return a direction to move. Any logic is valid — random, learned, hardcoded.

The Cold Case Files

Three real experiments from the 2000s changed neuroscience forever. The electrode recordings from those rat sessions are bundled into this codebase. Here is how the code finds, downloads, and uses that data — and why it matters for model validation.

Think of the Experiment classes as cold case detective files. Scientists wired real rats with electrodes, recorded exactly which neuron fired at which location, and those recordings became a permanent historical record. NeuralPlayground downloads, caches, and provides access to those files.

Experiments provide two things: real behavioral data trajectories (so your model follows the same path a real rat took) and real neural spike train data (so you can compare firing patterns). This comparison IS the scientific contribution.

Replaying a Real Experiment

When you pass use_behavioral_data=True, the arena stops being a generic box and becomes a precise replay of the original lab session. Here is what each line does — from the experimental arena usage pattern.

CODE 
from neuralplayground.agents import Stachenfeld2018
from neuralplayground.arenas import Sargolini2006

env = Sargolini2006(use_behavioral_data=True)
agent = Stachenfeld2018()

obs, state = env.reset()
for step in range(env.total_number_of_steps):
    action = agent.act(obs)
    obs, state, reward = env.step(action)
    agent.update()
PLAIN ENGLISH

import the successor representation brain model

import the arena that wraps the Sargolini experiment

Load a virtual replica of the Sargolini lab experiment, AND load the actual rat trajectory from disk (auto-downloads if not cached)

Create the brain model — this is the theory we want to test against real data

Place the agent at the same starting position a real rat used in the original experiment

Loop exactly as many steps as the real rat took in the experiment

Ask the model where to go next (or follow the real rat path if behavioral data overrides)

Let the model learn from this position, just as it would in any other arena

The key difference from a plain arena: env.total_number_of_steps is not a setting you choose — it is the exact length of the original rat recording. The loop ends when the replay ends.

Science at Scale

From one simulation to fifty — automated, logged, and reproducible

A single simulation tells you almost nothing scientifically. You need Weber2018 vs Stachenfeld2018, in multiple arenas, with 5 random seeds each, all results logged and comparable. Without SimulationManager you write 50 loops by hand. With it: three lines of code.

Think of it like a wine competition. Running one simulation is like tasting one wine — interesting, but scientifically meaningless. Good science requires a systematic tasting: same models, five different starting conditions, compared side-by-side with careful notes. SimulationManager is the sommelier who organizes the whole tasting.

N
The combinatorial explosion

N agents × M arenas × K seeds = every combination, automatically. SimulationManager turns a 50-line nested loop into a single configuration object.

✓
Failure-tolerant by design

If one run crashes, the manager logs the error, skips forward, and finishes the rest. Come back, call check_run_status(), and fix only what broke.

📁
Structured output from the start

generate_sim_paths() creates a clean folder hierarchy before the first run — so results are always organized and never overwritten.

Teaching the TEM to Want

How we bolted a reward signal onto an unsupervised brain model — and what changed inside the network

An Agent That Does Not Care

The original TEM is purely unsupervised. It explores, it builds an internal map, it learns to predict what it will see next. But it has no goal. Every location is equally interesting. Every step is equally worth learning from.

Real brains do not work this way. Hippocampal place cells fire more reliably near reward locations. Grid cell firing fields distort around goals. The dopamine system broadcasts a reward-prediction error that scales how strongly any given experience is consolidated into memory. The question is: can we add this to TEM without rebuilding it from scratch?

🎯
The Design Constraint

TEM is a large, carefully tuned unsupervised learner. The goal is to add reward sensitivity on top of it — a thin RL layer — without touching the core generative model or breaking the existing Hebbian memory dynamics. Three things need to change: the observation must carry the reward signal, the learning rate must respond to value, and value must propagate backwards from the reward location.

TD(0): The Simplest Reward Propagator

TD(0) maintains a table V where V[s] estimates the total future reward the agent expects to collect starting from state s. Every time the agent transitions from state s to state s', the estimate is nudged:

TD(0) update rule

error = r + γ · V[s'] − V[s]
V[s] = V[s] + α · error

r = reward at s (1 if reward state, else 0)  |  γ = discount (0.95)  |  α = step-size (0.1)

The critical insight: V values near the reward state are updated first, then over many rollouts the non-zero values propagate outward like ripples — states two steps from reward eventually get V ≈ γ², states three steps get V ≈ γ³, and so on. This is "backwards propagation" through the state space, implemented without any gradient or explicit graph traversal.

CODE — agents/whittington_2020.py · _update_td_values() 
for step_i in range(n_steps - 1):
    for env_i in range(self.batch_size):
        s      = history[step_i][env_i][0]
        s_next = history[step_i + 1][env_i][0]
        r      = 1.0 if s == self.reward_states[env_i] else 0.0
        td_err = r + gamma * self.V_table[env_i, s_next] - self.V_table[env_i, s]
        self.V_table[env_i, s] += alpha * td_err
PLAIN ENGLISH

for step_i in range(n_steps - 1) — Loop over every consecutive pair of timesteps in the 20-step rollout

s = history[step_i][env_i][0] — The current state ID for environment env_i at this step

r = 1.0 if s == self.reward_states[env_i] — Is this the reward location? If yes, r=1. If no, r=0. There is one reward per grid, placed at the centre state.

td_err = r + gamma * V[s_next] - V[s] — The surprise: how much better (or worse) was reality than our prediction? Positive error → V[s] was too low. Negative → too high.

V[s] += alpha * td_err — Nudge the estimate in the right direction by a small step. After thousands of rollouts, V[s] converges to the true discounted return from s.

Where does the reward state live?

Each environment gets one reward state, initialised in _init_td() as n_states // 2 — the centre of the discrete state space. Because TEM trains on 16 environments simultaneously, there are 16 independent value tables and 16 independent reward locations. The agent never teleports to the reward; it must walk there and accumulate experience, just like a real rat.

The Reward Channel: Extending X

The original TEM receives a one-hot observation X of shape (batch, 45). To let the network encode reward proximity, we append one extra dimension: the normalised TD value V(s), scaled so the maximum possible discounted return maps to 1.0.

The 46-dim vector enters the model's compression function f_c(). That function splits it: the first 45 dims go through the original two-hot lookup table (producing a 10-bit code), while the reward dim is appended directly, yielding an 11-dim compressed vector. Every subsequent operation in the model — temporal filtering, memory lookup, place-cell formation — now operates on this 11-dim representation, meaning reward proximity is literally woven into the place-cell pattern at every location.

CODE — whittington_2020_model.py · f_c() 
def f_c(self, decompressed):
    n_reward = self.hyper.get("n_td_reward", 0)
    if n_reward > 0:
        obs_part    = decompressed[:, :-n_reward]
        reward_part = decompressed[:, -n_reward:]
        two_hot = torch.stack(
            [self.hyper["two_hot_table"][i]
             for i in torch.argmax(obs_part, dim=1)], dim=0
        ).to(dtype=decompressed.dtype, device=decompressed.device)
        return torch.cat([two_hot, reward_part], dim=1)
PLAIN ENGLISH

obs_part = decompressed[:, :-n_reward] — Slice off the first 45 columns: the ordinary sensory one-hot observation

reward_part = decompressed[:, -n_reward:] — The last column: the normalised TD value for this state, between 0 and 1

two_hot = torch.stack([two_hot_table[argmax(obs)]...]) — Convert each row's one-hot observation to its 10-bit two-hot code, exactly as before

return torch.cat([two_hot, reward_part], dim=1) — Glue the 10-bit sensory code and 1-bit reward channel together into an 11-dim vector. This is what all downstream memory operations will see.

🔗
Why the Loss Function Also Changed

The TEM loss computes cross-entropy between predicted observations and the true observation label. That label comes from argmax(x). With X now being 46-dim, argmax could accidentally pick index 45 (the reward channel) if its value happened to be the largest. To prevent this, the loss now masks: argmax(x[:, :45]). The model predicts sensory observations only; the reward channel is not a prediction target.

Value-Scaled Learning Rate

Knowing that a location is near reward is useful — but the TEM's unsupervised losses treat all visited states identically. To make the network consolidate reward-proximal experiences more strongly, the learning rate passed to the Adam optimiser is scaled by the current batch's average TD value:

Effective learning rate

eff_lr = base_lr × (1 + td_lr_scale × mean_V_normalised)

base_lr = scheduled rate (decays over training)  |  td_lr_scale = 2.0  |  mean_V_normalised = mean(V_current_states) × (1 − γ)

At the start of training V is everywhere zero, so the effective rate equals the base rate — no change. As values build up near the reward, rollouts that happen to pass through high-value states get a larger gradient step. The multiplier is bounded: at maximum value (1.0 normalised), the rate is tripled (1 + 2.0 × 1.0 = 3). This is intentional — rare, valuable experiences should be consolidated more strongly.

CODE — agents/whittington_2020.py · update() 
curr_values = np.array([
    self.V_table[env_i, history[-1][env_i][0]]
    for env_i in range(self.batch_size)
])
norm_scale    = 1.0 - self.pars.get("td_gamma", 0.95)
avg_norm_value = float(np.mean(curr_values) * norm_scale)
effective_lr  = self.lr * (
    1.0 + self.pars.get("td_lr_scale", 2.0) * avg_norm_value
)
PLAIN ENGLISH

curr_values = np.array([V_table[env_i, last_state] ...]) — Collect the raw TD values for the final state of each environment in the batch — these are the most recently visited states

norm_scale = 1.0 - td_gamma — Multiply by (1 − γ) = 0.05. This scales the maximum possible value (1/(1−γ) ≈ 20) back down to 1.0, giving a normalised signal in [0, 1]

avg_norm_value = mean(curr_values) * norm_scale — Average across the 16 environments to get one scalar signal. High = the batch spent time near reward.

effective_lr = lr * (1 + td_lr_scale * avg_norm_value) — Scale the scheduled learning rate up. At zero value → no change. At full value → 3× the base rate.

What Actually Changed in the Network

Adding one reward dimension to the compressed observation cascades through TEM's architecture. Understanding the cascade requires knowing how the network's internal dimensions are related.

Before

  • n_x = 45  (one-hot observation)
  • n_x_c = 10  (two-hot bits)
  • n_x_f = 10  (per-frequency filtered obs)
  • n_p = n_g_sub × 10  (place cells)
  • sum(n_p) = 400  (total place cells)
  • M shape = (batch, 400, 400)

After (+TD)

  • n_x = 45  (unchanged)
  • n_x_c = 10  (unchanged)
  • n_x_f = 11  ↑ +1 reward dim
  • n_p = n_g_sub × 11  ↑ larger
  • sum(n_p) = 440  ↑ +40 cells
  • M shape = (batch, 440, 440)  ↑ larger memory

The change flows automatically from one line in whittington_2020_parameters.py: n_x_f = n_x_c + n_td_reward. Because every downstream dimension (n_p, W_tile, W_repeat, p_update_mask) is computed from n_x_f, they all resize without further edits. The two-hot table itself is unchanged — it still maps 45 observations to 10-bit codes. The reward channel bypasses that table entirely, joining the representation after compression.

🧩
What the Larger Memory Means

Each place cell is the outer product of a grid-cell code (abstract location) and the compressed sensory+reward vector. With 11-dim sensory+reward instead of 10-dim sensory, the outer product grows proportionally. The memory matrix M is now large enough to bind "where I am" to "what I see AND how much reward is here." In principle, the model can retrieve reward proximity purely from its internal map — without being explicitly told the reward location at inference time.

Three Changes, One Behaviour

The full TD modification touches exactly three files and introduces exactly four new parameters. Here is the complete picture:

whittington_2020_parameters.py

Adds n_td_reward=1, td_gamma=0.95, td_alpha=0.1, td_lr_scale=2.0. Changes n_x_f from n_x_c to n_x_c + n_td_reward. All downstream dimensions cascade automatically.

whittington_2020_model.py

Three surgical edits: f_c() splits the 46-dim input to route obs through two-hot and append the reward channel; b_x is resized from n_x_c to n_x_f[0] to match the wider compressed representation; loss() masks argmax to the first 45 dims so the reward channel does not corrupt prediction targets.

whittington_2020.py (the agent wrapper)

Three new methods: _init_td() builds the value table and places one reward per environment; _update_td_values() runs TD(0) over every rollout; _get_td_signals() returns normalised values to append to X. The existing update() method is extended to call all three and to scale the Adam learning rate by the current batch value.

What you can experiment with

  • td_gamma — Lower it (e.g. 0.5) for a short-horizon agent that only values nearby reward; raise it toward 1.0 for a long-horizon planner.
  • td_alpha — The step-size for value updates. Higher values converge faster but can oscillate.
  • td_lr_scale — Controls how much the model's learning rate amplifies near reward. Set to 0.0 to disable LR scaling while keeping the reward channel in X.
  • reward_states — Currently set to n_states // 2 (arena centre). Change _init_td() to place rewards at corners, multiple locations, or different positions per environment.
  • n_td_reward — Could be extended beyond 1 to encode richer reward signals (e.g. separate channels for different reward magnitudes).
