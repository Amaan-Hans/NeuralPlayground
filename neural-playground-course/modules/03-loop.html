The Heartbeat Loop

Four lines that run 20,000 times — and everything else supports them

Four Lines Rule the World

There is one piece of code that runs tens of thousands of times in every simulation. It is four lines long. Everything else in the framework exists to support those four lines.

A heartbeat has four phases: squeeze, pause, refill, relax. Miss one and the whole system stops. The reinforcement learning training loop has the same rhythm: observe, act, step, update. This four-beat pattern is universal — it appears in every RL system ever written.

The Loop, Line by Line

Every line in the core loop has one job. Here is exactly what each one does in plain English — from backend/training_loops.py, lines 29–41.

CODE 
obs, state = env.reset()
training_hist = []
obs = obs[:2]
for j in range(round(n_steps)):
    action = agent.act(obs)
    obs, state, reward = env.step(action)
    update_output = agent.update()
    training_hist.append(update_output)
    obs = obs[:2]
PLAIN ENGLISH

Reset the arena: randomize the starting position, clear all history from previous runs

Create an empty list to store learning statistics as training progresses

Keep only x,y coordinates — the arena returns extra data, but we only need position

Repeat this block for every step in the simulation (up to n_steps total moves)

Ask the brain model: given where you are, which direction do you want to move?

Execute that movement: check for walls, update position, return new location and reward signal

The brain model learns: adjust connection weights based on what it just experienced

Log the learning statistics for later analysis and debugging

Trim position data again for the next loop iteration

Notice the observation space slice obs[:2] appears twice: once before the loop, once at the end of each iteration. The arena hands back more data than the agent needs — that trim keeps everything consistent.

The Four-Beat Rhythm

Each beat has a single owner and a single responsibility. The agent owns beats 1 and 2 and 4. The arena owns beat 3. Neither can skip its turn.

1
OBSERVE

The agent receives its current x,y position from the arena. This is all it knows — no memory of past positions, no knowledge of walls, just a two-number coordinate.

2
ACT

The brain model computes which direction to move. In the base agent, this is a random Gaussian step. In Weber2018, it uses learned weights shaped by the hippocampus model.

3
STEP

The arena executes the move: checks for walls via validate_action(), updates position, calculates reward signal, and returns the new state.

4
UPDATE

The brain model learns. It adjusts its internal connections based on where it just was and what it experienced. In episode terms, this is where the neurons wire together.

🔬
The Base Agent Does Almost Nothing

The agent.update() method in AgentCore returns None. That is intentional — it is a placeholder that subclasses like Weber2018 override with real learning algorithms. The loop structure stays the same regardless.

Where Do You Plug In?

When you ask AI to add a feature, the answer is almost always "put it in the loop — but where?" The answer depends on what information that feature needs.

before the loop One-time setup: load a checkpoint, open a log file, initialize a renderer. Runs once.
after agent.act() You have the action but not yet the result. Useful for visualizing intended movement before it happens.
after env.step() ★ Most common insertion point. You have the new position and reward — ideal for logging, visualization, early stopping, and progress checks.
after agent.update() You have the learning stats from this step. Good for logging loss curves or saving checkpoints at intervals.
after the loop Post-processing: plot trajectories, save the final model, run summary statistics. Runs once.
